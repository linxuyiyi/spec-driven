# 质量验收标准 (Quality Gate)

> 测试步骤输出的质量评估标准 + Badcase 收集模板。没有质量闭环的交付叫自嗨。

---

## 质量评估维度

### 维度 1：完整性 (Completeness) - 权重 30%

| 等级 | 标准 | 得分 |
|------|------|------|
| A | 步骤覆盖端到端全流程，含前置条件、执行步骤、验证点、后置处理 | 90-100 |
| B | 步骤基本完整，但缺少前置或后置处理 | 70-89 |
| C | 缺少关键步骤（如验证点缺失） | 50-69 |
| D | 流程不完整，无法直接执行 | <50 |

**检查清单**：
- [ ] 有环境重置/前置条件步骤
- [ ] 有完整的执行步骤序列
- [ ] 有验证点（至少 3 个）
- [ ] 有后置处理（如环境清理）

---

### 维度 2：准确性 (Accuracy) - 权重 30%

| 等级 | 标准 | 得分 |
|------|------|------|
| A | 参数值合理且带单位，协议引用正确，预期结果明确可验证 | 90-100 |
| B | 参数基本正确，但部分缺少单位或预期结果模糊 | 70-89 |
| C | 参数有明显错误或遗漏 | 50-69 |
| D | 参数错误导致无法执行 | <50 |

**检查清单**：
- [ ] 所有参数带单位 (MHz/dBm/ms 等)
- [ ] 参数值在合理范围内
- [ ] 协议引用正确 (如 A3 触发条件)
- [ ] 预期结果可量化验证

---

### 维度 3：可执行性 (Executability) - 权重 25%

| 等级 | 标准 | 得分 |
|------|------|------|
| A | 步骤粒度适中，每个步骤对应一个 AW 函数，可直接映射 | 90-100 |
| B | 部分步骤需要进一步拆解 | 70-89 |
| C | 步骤过于粗粒度或过于细碎 | 50-69 |
| D | 无法映射到 AW 函数 | <50 |

**检查清单**：
- [ ] 每个步骤是原子操作
- [ ] 步骤名用动词 + 名词格式
- [ ] 步骤可映射到 AW 函数
- [ ] 无歧义描述

---

### 维度 4：覆盖度 (Coverage) - 权重 15%

| 等级 | 标准 | 得分 |
|------|------|------|
| A | 覆盖正常流程 + 至少 2 个变体场景 + 负例场景 | 90-100 |
| B | 覆盖正常流程 +1 个变体场景 | 70-89 |
| C | 仅覆盖正常流程 | 50-69 |
| D | 正常流程覆盖不全 | <50 |

**检查清单**：
- [ ] 正常流程覆盖
- [ ] 至少 1 个变体场景
- [ ] 至少 1 个负例场景
- [ ] 边界条件覆盖

---

## 综合评分公式

```
总分 = 完整性×30% + 准确性×30% + 可执行性×25% + 覆盖度×15%
```

| 总分 | 等级 | 处置 |
|------|------|------|
| 90-100 | A | 直接入库，标注为"推荐用例" |
| 70-89 | B | 修改后入库 |
| 50-69 | C | 需要返工，重新扩写 |
| <50 | D | 不可用，责任人复盘 |

## Badcase 收集模板

当 AI 输出的测试步骤质量不达标时，用此模板收集并反馈。

### Badcase 记录表

```markdown
## Badcase #___

**发现日期**: 2026-XX-XX
**发现人**: @xxx
**涉及场景**: (如：A3 切换测试)
**Pattern 文件**: `patterns/handover-a3.md`

### 问题描述
(描述 AI 输出的步骤存在什么问题)

### 问题分类
- [ ] 完整性缺失 (缺少步骤)
- [ ] 准确性问题 (参数错误/协议引用错误)
- [ ] 可执行性问题 (步骤无法映射 AW 函数)
- [ ] 覆盖度不足 (缺少变体/负例)
- [ ] 其他：_______

### 原始输出
```markdown
(粘贴 AI 输出的有问题的步骤)
```

### 期望输出
```markdown
(粘贴或描述正确的步骤应该是什么样)

### 根因分析
(分析 AI 为什么会输出错误内容)
- Pattern 描述不清晰？
- 意图识别错误？
- 参数默认值缺失？
- 其他：_______

### 修复动作
- [ ] 更新 Pattern 文件
- [ ] 更新 SKILL.md 意图映射
- [ ] 更新参数默认值
- [ ] 其他：_______

### 修复日期
_______

### 验证结果
- [ ] 修复后验证通过
- [ ] 仍需优化
```

## Badcase 示例

### Badcase #001 - 参数缺少单位

**发现日期**: 2026-03-27
**涉及场景**: A3 切换测试

**问题描述**: AI 输出的步骤中参数没有带单位

**原始输出**:
```markdown
| 5 | A3 测量配置 | 配置 A3 事件测量 | threshold=-95, hysteresis=3, ttt=240 | 测量配置生效 |
```

**期望输出**:
```markdown
| 5 | A3 测量配置 | 配置 A3 事件测量 | threshold=-95dBm, hysteresis=3dB, ttt=240ms | 测量配置生效 |
```

**根因分析**: Pattern 文件中参数字段没有明确要求带单位

**修复动作**:
- [x] 更新 `patterns/handover-a3.md` 参数字段说明
- [x] 在 SKILL.md 中强化"参数带单位"规则

---

### Badcase #002 - 验证点缺失

**发现日期**: 2026-03-27
**涉及场景**: 小区建立测试

**问题描述**: 输出步骤缺少验证点列表

**原始输出**:
```markdown
## 测试步骤
(步骤表格)
(没有验证点列表)
```

**期望输出**:
```markdown
## 测试步骤
(步骤表格)

## 验证点
1. 小区状态在 5s 内变为 ACTIVE
2. SIB1 周期发送正常
3. UE 能够成功附着
```

**根因分析**: SKILL.md 中输出格式示例没有强调验证点必填

**修复动作**:
- [x] 在 SKILL.md 中明确验证点为必填项

## 质量复盘机制

每月末对 Badcase 进行复盘，回答四步法：

1. **回顾目标**: 本月 AI 输出的测试步骤合格率目标是多少？实际是多少？
2. **评估结果**: 哪些 Badcase 重复出现？根因是什么？
3. **分析原因**: 是 Pattern 质量问题、SKILL.md 规则问题、还是 AI 理解问题？
4. **沉淀规律**: 哪些规则可以固化为检查清单？哪些 Pattern 需要更新？

---

## 持续改进飞轮

```
Badcase 收集 → 根因分析 → Pattern/SKILL 优化 → 质量提升 → Badcase 减少
     ↑                                                              │
     └──────────────────────────────────────────────────────────────┘
```

**度量指标**:
- Badcase 数量 (周维度)
- Badcase 修复率
- 重复 Badcase 占比
- 平均质量评分趋势
